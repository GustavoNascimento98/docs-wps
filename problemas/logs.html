    Coletando Logs

    Os logs são o recurso mais importante do desenvolvimento para identificar e resolver bugs do Parking Plus encontrados em campo.

    É importante que técnicos e analistas de suporte saibam como ativar a geração destes logs, coletar os mesmos, e fazer uma análise inicial do seu conteúdo.

    Localizando o wpslogger.ini

    Todas as aplicações do Parking Plus respeitam o arquivo de configuração wpslogger.

    A localização e nome exato do arquivo varia dependendo do sistema operacional e da aplicação em questão:

    Situação Localização
    Aplicação Desktop Windows C:\WINDOWS\wpslogger.ini
    Serviço Windows C:\WINDOWS\wpslogger.ini
    DLL Windows Pasta do usuário.
    Digite no Windows Explorer:
    %HOMEDRIVE%%HOMEPATH%\wpslogger.ini
    Serviço Linux (RPM) /etc/wpslogger.conf
    Serviço Linux (Docker) /etc/ParkingPlus/NOMESERVICO/wpslogger.conf
    Substitua NOMESERVICO pelo nome do serviço em questão (e.g. PPCS).
    Boot Remoto Verifique o arquivo /tftpboot/config.conf.network no servidor de boot remoto. Um local convencionado é /tftpboot/config.wpslogger.conf.

    Configurando

    Após localizar o arquivo correto para a sua situação, basta edita-lo conforme desejado. A chave mais importante é a Enable, na seção LOGGER.

    Dica

    Não é necessário reiniciar a aplicação após alterar as configurações; O Parking Plus detecta alterações na configuração automaticamente após alguns segundos. Entretanto, certas mensagens só são geradas nada inicialização da aplicação e podem ser necessárias para identificar um bug; na dúvida pergunte ao desenvolvedor que solicitou os logs.

    O arquivo wpslogger.ini têm o seguinte formato:

    [LOGGER]
MainThreadOnly=0
Enable=1
Type=LOCAL
Level=INFO

[DC]
EXCLUDE=
FORCE=
INCLUDE=

[LOCAL]
Path=C:\WPS Brasil\logs\

[REMOTE]
Host=servidor
Port=6070

    LOGGER

    MainThreadOnly

    Em geral, não deve ser alterada. Utilize sempre o valor 0. Esta configuração é para uso interno dop time de desenvolvimento.

    Indica se o log de threads secundarias deve ser incluido (valor 0) ou omitido (valor 1) nos logs.

    Enable

    Indica se logs devem ou não ser gerados. Use 1 para gerar logs, 0 para não logar nada.

    Type

    Indica como os logs devem ser gerados. Deve ser um dos valores abaixo:

    REMOTE

    Os logs são enviados via UDP para um servidor de logs remoto. Este formato apresenta risco de perdas, mas é a única opção viável para estações com boot remoto.

    CONSOLE

    Os logs serão escritos no STDOUT da aplicação.

    Este é o formato preferido para serviços Docker e/ou RPMs no CentOS 7, pois assim os logs serão coletados pelo journal.

    LOCAL

    Os logs serão salvos em arquivos de texto, em pasta configurável. O log rotaciona diariamente, mas não existe nenhum mecanismo para apagar ou comprimir logs antigos.

    Aviso

    Sempre que possível, evite usar este tipo de log. Ele pode causar esgotamento do espaço em disco e levar a, no pior dos casos, uma parada total da garagem.

    Level

    Indica qual a quantidade de logs que será gerada. Deve ser um dos valores abaixo, ordenados do menor conteúdo para o maior:

    • FATAL somente erros graves.
    • ERROR qualquer erro.
    • WARN qualquer erro e avisos importantes.
    • INFO qualquer erro e qualquer aviso.
    • DEBUG erros, avisos e mensagens para desenvolvedores.
    • TRACE todas as mensagens possíveis; detalha todos os passos da aplicação.

    DC

    Uso interno. Permite filtrar mensagens pelo contexto em que são geradas.

    Só altere se for solicitado por um desenvolvedor.

    LOCAL

    Só têm efeito quando LOGGER/Type for LOCAL. Configura logs em arquivo.

    Path

    Pasta do sistema na qual os logs serão gerados. Não é possível configurar o nome dos arquivos.

    REMOTE

    Só têm efeito quando LOGGER/Type for REMOTE. Configura destino dos logs remotos.

    O servidor de destino deve estar rodando uma instância do PPLF. Em servidores legados, ao invés do PPLF utilize o PPLS.

    Host

    Nome (DNS) ou endereço IP do servidor de logs.

    Port

    Número da porta UDP na qual o PPLS/PPLF está escutando no servidor remoto. É importante que esta porta esteja aberta no firewall do servidor.

    Localizando os logs

    Após configurar os logs a forma de coleta depende de alguns fatores, detalhados abaixo.

    Dica

    Em todos os casos, recomendamos que os logs sejam comprimidos (i.e. zip, tgz, etc) para facilitar seu envio. Por serem arquivos de texto com muitas repetições, mesmo a compressão básica reduz o tamanho em 90% ou mais.

    Log Console

    O log console é o log ideal para serviços Linux, sejam eles RPM ou Docker. Para demais aplicações, especialmente Windows, o log console é difícil / impossível de ser coletado.

    Para serviços com log console, os logs podem ser coletados via journal:

    Tipo Exemplo
    RPM $ journalctl -a -u NOMESERVICO.service
    Docker $ journalctl -a CONTAINER_TAG=ParkingPlus:NOMESERVICO

    Log Remoto

    Refira-se à documentação do PPLF (ou PPLS em servidores legados).

    No PPLF, os logs estarão no journal do servidor.

    Log Local

    Este é o tipo mais fácil de ser localizado. Os logs estarão na pasta indicada por LOCAL/Path do arquivo de configuração dos logs. O nome dos arquivos sera a data na qual eles foram gerados. I.e. se hoje for 13/10/2021, o log gerado hoje será 20211013.wpslog.

    É importante notar que logs locais (como o nome indica) sempre serão salvos no mesmo computador no qual a aplicação esta executando. Isto quer dizer que:

    • Para estações com boot remoto, logs locais estão na memória da estação e serão perdidos caso a estação seja reiniciada.
    • Para serviços Docker, dependendo da pasta na qual os logs forem salvos eles não podem ser acessados fora do container. Sempre use logs CONSOLE para docker.

    Por estes e outros motivos, o log local deve ser a última opção. Ele é aceitável apenas para aplicações Desktop Windows, onde os impactos de suas limitações são menores.
